## 2. Logic and Proofs

### Boolean Algebra Homomorphism Claims

**Union preservation (F(A cup B) = F(A) | F(B))**: CORRECT. The proof in the blog post is thorough and correct. The paper (Sec 4.1 Definitions 4-6, used implicitly) relies on the same argument: OR is idempotent (h(x) | h(x) = h(x)), so F(A cup B) = OR over all h(x) for x in A cup B = (OR over A) | (OR over B) = F(A) | F(B).

**Intersection preservation (F(A cap B) = F(A) & F(B))**: CORRECT for the specific representation. Given F(S) = h(x1) | ... | h(xk), intersection in the bit domain uses AND. The proof works because a bit is set in F(A) & F(B) iff it is set in both, which happens iff elements in both A and B hash to that position. However: this is an APPROXIMATION, not an exact homomorphism. F(A) & F(B) may have extra bits set (false positives in "intersection"). The paper does not fully acknowledge this in the homomorphism section -- it presents F as an exact homomorphism for union/intersection, then later discusses errors. This is technically correct but could be clearer.

**Complement non-preservation**: CORRECT. The proof (Thm 6 in paper, "Disproof of Third Property" in blog post) is sound: F(~A) = 1^n for infinite ~A (pigeonhole), but ~F(A) != 1^n for finite A.

### Size-Dependent FPR

**Membership FPR (Thm 4)**: The derivation is correct. After k insertions via OR, each bit position is set with probability 1 - 2^{-k}. For a random query h(x), each bit is set with probability 1/2. A false positive requires every bit in h(x) that is 1 to also be 1 in F(S). The per-bit probability is 1 - (1/2)(2^{-k}) = 1 - 2^{-(k+1)}, and independence across n bits gives (1 - 2^{-(k+1)})^n.

**Subset FPR (Thm 5)**: Correct derivation. For S1 subseteq S2 to false-positive, every bit set in F(S1) must be set in F(S2). Bit j is set in F(S1) with prob 1-2^{-k1} and unset in F(S2) with prob 2^{-k2}, giving per-bit non-refutation probability 1 - (1-2^{-k1})*2^{-k2}.

**Space complexity (Thm 7)**: Correct. n = O(2^k) follows directly from solving (1 - 2^{-(k+1)})^n = epsilon for n.

### Bit-Rate Formula (Corollary 1)

**ISSUE**: The approximation step is sloppy. The corollary states:

> b(n, epsilon) = log2(epsilon) / (n * alpha(n))

where alpha(n) = 1 - 2^{-(n+1)} is the "per-bit occupancy probability after n insertions." The variable naming is confusing (n is used both for number of elements and hash size elsewhere). More importantly, the approximation "log2(alpha(n)) ~ alpha(n) - 1 ~ -alpha(n) for small deviations" is wrong -- alpha(n) is close to 1 for large n, so log2(alpha(n)) ~ (alpha(n)-1)/ln(2), not just -alpha(n). This formula needs correction.

### Composition Accumulation (Corollary 2)

States: "For n operations each with symmetric error rate epsilon, the accumulated error is bounded by n*epsilon - O(epsilon^2)."

This is approximately correct for OR composition: after n operations, alpha = 1 - (1-epsilon)^n ~ n*epsilon for small epsilon. The O(epsilon^2) correction is from the inclusion-exclusion expansion. But the corollary is vague about which operations are being composed. For AND, the FPR actually decreases (multiplicative), so the statement only applies to OR/union chains.

## 3. Alignment with Cipher Map Framework

### Hash Construction = Batch Cipher Map

The hash transformation T_k(v) = H(k || v) is precisely a batch cipher map: the trusted machine applies it to all values with the secret key k, producing opaque bit strings. This maps directly to:
- **Totality**: H is a total function
- **Representation uniformity**: Random oracle gives uniform outputs
- **Correctness**: FPR = 2^{-n} per element, FNR = 0

### Boolean Operations = Online Cipher Maps

The bitwise operations (|, &, ~, ^) on hash values are online cipher maps: they operate on encoded data without the key. The untrusted machine performs these operations blindly. Error propagation rules characterize the correctness degradation.

### Missing Connection

The paper does NOT explicitly connect to the cipher map formalism. It uses "HBOS" terminology instead. This should be reconciled with the cipher-maps-unified paper. The natural framing:
- TrapdoorFactory.create() = the encoding cipher map (batch)
- BooleanSet.__or__(), __and__() = composition cipher maps (online)
- BooleanSet.contains() = the decoding step (returns Bernoulli Boolean)

## 4. Experiments

All benchmarks are design-only. The paper is transparent about this (Table 1 footnote). The complexity analysis is sound:
- Hash creation: O(1) per element (single SHA-256 call) -- microsecond-scale is realistic
- Set operations: O(n) where n is number of elements -- linear scan, so the projected times are plausible
- Membership: O(1) -- single hash comparison

The scalability claim "throughput remains constant for small sets and decreases logarithmically for large sets due to cache effects" is unsupported speculation with no data.

The comparison table (Table 2) is reasonable but should note that HBOS provides weaker security guarantees than FHE/MPC (preimage-based vs. semantic/simulation-based).

## 6. Specific Issues

### Terminology: "Oblivious" Is Misleading

The paper title uses "privacy-preserving" but the text repeatedly says "oblivious" and "Hash-Based Oblivious Sets." In cryptography, "oblivious" has a specific meaning (ORAM, oblivious transfer) implying access-pattern hiding. HBOS does NOT hide access patterns. Remark 5 acknowledges this but the terminology persists throughout. This will confuse cryptography reviewers.

**Recommendation**: Replace "oblivious" with "trapdoor" or "hash-hidden" throughout. Keep the ORAM comparison in a remark for clarity, but do not call the system "oblivious."
