## Review Summary

**Overall Assessment**: The paper has made real progress since the previous review: the three CRITICAL issues identified earlier have been addressed with varying degrees of success, and the proof infrastructure is now more substantive. However, four of the original MAJOR issues remain unresolved, the paper still contains a structurally broken "theorem" (Thm 3.5), the functoriality proof still has a well-definedness gap that is now more explicitly acknowledged but still not resolved, and obliviousness is still undefined formally. The paper is closer to submittable but is not yet ready for a rigorous venue.

**Recommendation**: major-revision

**Strengths**:
1. The split of the old Def 2.2 into Def 2.2 (Cipher Functor) and Def 2.3 (Lifted Operation) is the right move. The explicit formula for the lifted operation via decode-compute-reencode is now present and substantially improves the paper.
2. Thm 2.1 (Cipher Monoid Properties) now has a real proof that works from the explicit construction. The closure and identity steps are valid. The associativity handling (showing equality modulo decode-equivalence and flagging the limitation) is honest.
3. The reordering in Section 6 -- lower bound first (Thm 6.1), then space complexity (Thm 6.2) -- is correct and follows the fix for CRITICAL Issue 2.

5. The Corollary (eta=0 case) correctly connects the general bound to the abstract's claim.
6. Example 2.2 has been corrected to use (Z_p*, cdot) with ElGamal citation, resolving CRITICAL Issue 3.
7. Thm 7.5 (Cipher Composition) remains the paper's strongest result with a correct proof.
8. The Remark following Thm 2.1 (quotient monoid perspective) is a substantive clarification that adds genuine value.

**Weaknesses**:
1. Obliviousness is still never formally defined. This is the most consequential remaining gap for publication at a cryptography venue.
2. Thm 3.5 (Bernoulli Entropy) is labeled a theorem but states only a formula that is dimensionally inconsistent and is given without proof. It remains broken.

## Major Issues

### Issue 1: Obliviousness Is Still Not Formally Defined (UNRESOLVED from prior review)
**Severity**: MAJOR
**Location**: Def 4.1 (Cipher Map, lines ~441-456), Thm 7.3 (Bernoulli Obliviousness, lines ~689-691), Prop 4.2 (Representation Indistinguishability, lines ~481-486)
**Problem**: The prior review identified this as MAJOR Issue 7, and it has not been addressed. The paper still defines obliviousness via three informal bullets in Def 4.1: "mappings only revealed through direct evaluation," "domain cannot be efficiently enumerated," "unmapped inputs behave as random oracles." These are not a formal definition.

Thm 7.3 states "A Bernoulli approximation with symmetric confusion matrix and random oracle behavior for undefined inputs satisfies obliviousness conditions" -- this is circular (it says the approximation satisfies conditions without naming which conditions).

Prop 4.2 states representations are "computationally indistinguishable without knowledge of the encoding secret" -- this does not specify: a security parameter, a bound on adversary running time, or what the adversary's input and output are. This is not a formal security statement.

The paper targets applications including encrypted search, MPC, and ORAM-replacement. Any of these communities will require a formal definition. Without one, every "oblivious" theorem in the paper is a claim without content.

**Suggestion**: Add a formal definition at the start of Section 4, before Def 4.1. Standard options: (a) Access-pattern indistinguishability: A data structure is oblivious if for any two query sequences Q and Q' of the same length, the induced memory access patterns are computationally indistinguishable. (b) Semantic security style: No PPT adversary can distinguish which of two possible inputs generated an observed cipher representation sequence, with negligible advantage. The choice depends on what the paper's constructions actually achieve. All subsequent "obliviousness" theorems must then be restated with respect to this definition and proved from it.

### Issue 2: Thm 3.5 (Bernoulli Entropy) Is a Formula, Not a Theorem, and the Formula Is Dimensionally Wrong (UNRESOLVED from prior review)
**Severity**: MAJOR
**Location**: Theorem 3.5, lines ~426-430
**Problem**: The prior review identified this as MAJOR Issue 6. It has not been fixed. The current text states:

  "The entropy of a Bernoulli model with confusion matrix Q is: H(Bernoulli) = -sum_{i,j} q_{ij} log q_{ij}"

This formula is incorrect as stated. The confusion matrix Q has rows summing to 1 (each row is a conditional distribution), so sum_{i,j} q_{ij} = n (number of latent values), not 1. The formula -sum_{i,j} q_{ij} log q_{ij} is therefore not a valid entropy unless implicitly divided by n (which would give the average of row entropies) or unless a prior distribution over latent values is specified (which would give conditional entropy). The distinction matters: the formula as written double-counts the normalization.

Additionally, this is labeled a "Theorem" but has no proof and could not have a proof because it defines a quantity rather than stating a provable claim.

**Suggestion**: Replace with a Definition labeled "Bernoulli Conditional Entropy": given a prior distribution (pi_i) over latent values, define H(obs | latent) = -sum_i pi_i sum_j q_{ij} log q_{ij}. Alternatively, define the per-row entropy H_i = -sum_j q_{ij} log q_{ij} and the model entropy as the pi-weighted average. Remove the Theorem label. If a theorem is intended (e.g., that this quantity bounds the information leakage), state and prove that theorem separately.

### Issue 3: Functoriality Proof Has an Acknowledged but Unresolved Well-Definedness Gap (PARTIALLY ADDRESSED from prior review)
**Severity**: MAJOR
**Location**: Theorem 7.4 proof, lines ~719-734; Remark following Thm 2.1, lines ~238-240
**Problem**: The prior review identified this as MAJOR Issue 5. Progress has been made: the Remark after Thm 2.1 now explicitly states that the cipher monoid is most naturally viewed as a monoid on equivalence classes c_A S / sim, and that decode descends to an isomorphism. This is the right framing.

However, the proof of Thm 7.4 still defines the lifted homomorphism as cipher_A phi(c) = encode_2(phi(decode_1(c)), k) "for some canonical choice of representation index k." The proof then claims "Identity preservation is immediate" and shows encode(decode(c), k) acts "as the identity on c_A S up to representation choice." But:

(a) If c_A S is the raw set of representations (not the quotient), then encode(decode(c), k) != c in general (when c was encoded with a different index than k). The lifted identity is not the set-theoretic identity.

(b) If c_A S is the quotient c_A S / sim, then the claim is correct -- but the proof must then also restate Thm 7.4 in terms of the quotient category, not just say "up to representation choice."

(c) For composition preservation: the proof shows cipher_A(psi circ phi)(c) = encode_3((psi circ phi)(decode_1(c)), k) and (cipher_A psi circ cipher_A phi)(c) = encode_3(psi(phi(decode_1(c))), k), and equates them. This step uses the same index k for both applications of encode_3, which is only valid if the canonical k is the same in both cases. This must be justified.

The paper is inconsistent: Thm 7.4 claims c_A is a functor on "the category of monoids" (with morphisms being ordinary monoid homomorphisms), but the proof only gives functoriality up to equivalence. These are not the same claim.

**Suggestion**: Choose one of two paths:
(a) Work in the quotient category where objects are c_A S / sim and morphisms are well-defined on equivalence classes. Restate Thm 7.4 for this category. The proof then becomes clean.
(b) Introduce a fixed, computable "canonical representative" function rep: S -> c_A S such that decode(rep(s)) = s and define the functor using rep(decode(c)) instead of encode(decode(c), k) for some arbitrary k. This makes the functor a set-theoretic functor on the original category of monoids (with c_A S as the carrier), not just a functor up to equivalence.

### Issue 5: Thm 6.4 (Collision Resistance) Has No Proof
**Severity**: MAJOR
**Location**: Theorem 6.4 (Collision Resistance), lines ~661-668; Appendix A.2, lines ~900-909
**Problem**: This was flagged as a security theorem in the prior review (related to MAJOR Issue 7 on obliviousness, but technically separate). Thm 6.4 makes three claims:
1. Finding collisions requires Omega(2^{m/2}) operations.
2. Inverting the encoding requires Omega(2^m / |S|) operations.
3. The observable distribution is computationally indistinguishable from uniform.

None of these three claims has a proof in the main body. The appendix (A.2) contains a proof sketch that is incomplete: it sets up a reduction where algorithm B extracts a hash collision from adversary A's internal state, but does not explain the extraction mechanism. "B uses A's internal state to extract a hash collision" is not a valid step in a security reduction -- you cannot generally read an adversary's internal state.

Claim 3 (computational indistinguishability from uniform) requires the hash function h to be a pseudorandom function, which is a stronger assumption than random oracle (the latter is a non-standard model). This distinction is not made.

Claim 1 follows immediately from the birthday bound for random oracles (standard result) and could be proved or cited in one line. Claims 2 and 3 require more careful argument.

**Suggestion**: For Claim 1: cite the birthday bound directly (it is a standard result). For Claim 2: the preimage resistance of a random oracle follows immediately from counting: the probability of inverting a specific hash output is 2^{-m}, so Omega(2^m / |S|) queries are needed to find an input hashing into the valid set for a given element s. Write this out explicitly. For Claim 3: either (a) restrict the claim to random oracles and prove that the distribution of encode(s, k) for random k is uniform over c_A S under the construction's assumptions, or (b) restate using pseudorandom functions and cite an appropriate PRF assumption.

## New Issues Identified in This Review

### New Issue 1: Thm 6.2 (Space Complexity) Proof Step 3 Has a Logical Gap
**Severity**: MAJOR
**Location**: Theorem 6.2 proof, Step 3 (False positive control), lines ~624-628
**Problem**: The proof argues: "For non-members x not in {x_1,...,x_n}, the hash h(l) XOR h(x) is uniformly distributed under the random oracle model. The false positive rate epsilon is the probability this random string decodes to any valid output." This argument has a subtle gap: it assumes that the seed l (found by the construction for the stored elements) does not correlate with the non-member x. Under the random oracle model, h(x) for x not in the stored set is indeed independent of h(l) and of h(x_i) for all stored x_i -- but only if x is distinct from l and all x_i, which holds by assumption. So this step is almost right. The actual gap is more subtle: the construction chooses l specifically so that h(l) XOR h(x_i) decodes correctly for all stored x_i. This means l is not uniformly distributed -- it is the first value satisfying a set of constraints. For non-member x, the value h(l) XOR h(x) is not guaranteed to be uniform in the strict sense; it is uniform conditional on the event that the construction succeeded, which is a joint event. The proof needs to show this conditioning does not bias the distribution of h(l) XOR h(x) for non-members. Under the random oracle model this follows from the independence of h(x) for distinct x values -- but this argument needs to be made explicit.
**Suggestion**: Add one sentence: "Since x differs from l and all stored x_i, h(x) is an independent uniform bit string under the random oracle model, and its XOR with h(l) is also uniform regardless of the value of l." Then cite this independence as the key property of the random oracle model.

### New Issue 2: Thm 2.1 Proof Shows Only Decode-Equality, Not Full Monoid Equality in c_A S
**Severity**: MAJOR
**Location**: Theorem 2.1 proof, Associativity step, lines ~222-231
**Problem**: The associativity proof (for the lifted operation) shows:
  decode((x c_A* y) c_A* z) = decode(x c_A* (y c_A* z))
and then states "Since decode maps both sides to the same element of S, the results are representations of the same underlying value." The proof then acknowledges that "in general, associativity holds up to the equivalence x ~ y <=> decode(x) = decode(y), which is the operationally relevant notion."

This is honest but it means Thm 2.1 as stated is false: the theorem claims (c_A S, c_A*, encode(e,0)) is a monoid, but associativity (x c_A* y) c_A* z = x c_A* (y c_A* z) as an equation in c_A S does not hold in general for arbitrary selector rho. The theorem should be stated as: "(c_A S / ~, c_A*, [encode(e,0)]) is a monoid" or with the additional hypothesis that rho is a "consistent selector" (as the proof defines). The theorem statement and the proof conclusion are currently misaligned.

Additionally, the identity axiom proof shows:
  decode(encode(e,0) c_A* x) = e * decode(x) = decode(x)
but again this only shows decode equality, not equality in c_A S. The identity is encode(e,0), and for this to be a left identity we need encode(e,0) c_A* x = x in c_A S, not merely decode(encode(e,0) c_A* x) = decode(x).

**Suggestion**: The theorem statement must be corrected in one of two ways:
(a) State the theorem for the quotient monoid (c_A S / ~, c_A*, [encode(e,0)]) and prove associativity and identity in the quotient. The proof as written already proves this.
(b) Add "consistent selector" as a hypothesis: say rho is consistent if the selector output depends only on the decoded value of the operands (not their specific representations). Under this hypothesis, prove the stronger statement that c_A S is a monoid as a set. The proof would need to show that consistent rho implies (x c_A* y) c_A* z = x c_A* (y c_A* z) in c_A S, not just under decode.

This is a genuine mathematical error in the theorem statement, not just a presentation issue.

### New Issue 3: Example 2.2 (Multiplicative Group Cipher) Has a Mathematical Error
**Severity**: MAJOR
**Location**: Example 2.2, lines ~332-341
**Problem**: The example claims that the cipher representations of a in Z_p* are encode(a, j) = g^{k + j(p-1)} for a = g^k. By Fermat's little theorem, g^{p-1} = 1 in Z_p*, so g^{k + j(p-1)} = g^k * (g^{p-1})^j = g^k * 1 = g^k = a. This means encode(a, j) = a for all j -- there is only one representation of a in Z_p*, not multiple. The "representation multiplicity" claim fails: all encoding indices j give the same element g^k.

The key property of the cipher functor (that distinct indices k yield distinct representations) is violated in this example as stated. The authors need a different construction to get multiple representations in Z_p*.

**Suggestion**: One approach: use a larger ambient group. For example, work in Z_{p^2}* (the multiplicative group mod p^2), where elements of the subgroup of order p-1 have multiple representations via the p-1 cosets of the subgroup. Alternatively, use an ElGamal-style construction where the representation of a is the pair (g^r, a * h^r) for random r, giving multiple representations indexed by r. The ElGamal ciphertext is the natural multi-representation structure for Z_p*. This would actually strengthen the section by connecting directly to the cited ElGamal reference.

### New Issue 4: Def 3.1 (Bernoulli Approximation) Requires the Error Variables to Be Independently Identically Distributed, But This Is an Unstated Assumption
**Severity**: MINOR
**Location**: Definition 3.1, lines ~352-356
**Problem**: The definition says the errors {e(x) : x in X} are "a priori statistically independent random variables." This is a significant modeling assumption (that errors are independent across inputs) that is stronger than needed for many results and weaker than needed for others. The assumption is stated as part of the definition rather than as a hypothesis, which means every result derived from Bernoulli approximations implicitly assumes independent errors. For the confusion matrix representation (Def 3.2) and the Cipher-Bernoulli correspondence (Thm 4.1), this independence assumption is crucial. For the composition theorem (Thm 7.5), independence across layers is assumed separately.

Additionally, e(x) is described as a random variable, but in the subsequent use it is treated as a fixed parameter (the error rate for input x). These are different: a random error rate vs. a fixed error probability.

**Suggestion**: Clarify whether e(x) is (a) the probability of error for input x (a fixed number in [0,1]), or (b) the random indicator variable that is 1 when an error occurs. In the definition as written ("Pr[p*(x) != p(x)] = e(x)") it is natural to read e(x) as a fixed probability -- then calling it a "random variable" is incorrect. The statement should read: "with error probability e(x) in [0,1], where the error events {p*(x) != p(x)} for different x are independent."

### New Issue 5: Thm 7.3 (Bernoulli Obliviousness) Is Stated but Not Proved
**Severity**: MINOR
**Location**: Theorem 7.3, lines ~689-692
**Problem**: This theorem states "A Bernoulli approximation with symmetric confusion matrix and random oracle behavior for undefined inputs satisfies obliviousness conditions." No proof is given. Even setting aside the problem that obliviousness is not formally defined (Issue 1 above), the claim is not self-evident. Why does a symmetric confusion matrix imply indistinguishability? The connection to the informal obliviousness bullets in Def 4.1 is not shown.
**Suggestion**: After fixing Issue 1 (formal obliviousness definition), prove this theorem from the definition. For access-pattern obliviousness: show that the sequence of cipher representations produced by the Bernoulli map for any input sequence is computationally indistinguishable from a sequence for any other input sequence (conditioned on same length). For semantic security: show the symmetric confusion matrix implies the adversary's posterior over the latent equals its prior.

### New Issue 6: Thm 7.4 Uses Two Different Encoding Functions with the Same Canonical k
**Severity**: MINOR
**Location**: Theorem 7.4 proof, line ~730
**Problem**: In the composition preservation step, the proof writes:
  cipher_A(psi circ phi)(c) = encode_3((psi circ phi)(decode_1(c)), k)
  (cipher_A psi circ cipher_A phi)(c) = encode_3(psi(phi(decode_1(c))), k)
The second line requires computing encode_3(..., k) but in cipher_A phi, the intermediate step goes through encode_2(..., k) then decode_2. The argument assumes decode_2(encode_2(phi(decode_1(c)), k)) = phi(decode_1(c)), which holds by the decode axiom, and then encode_3(psi(phi(decode_1(c))), k) = encode_3(..., k). The k used in the final encode_3 must be the same in both chains. The proof implicitly assumes this consistency but does not state it.

More critically: in the composition (cipher_A psi circ cipher_A phi)(c), the index used for the intermediate encode_2 in cipher_A phi may differ from the k used in the outer encode_3 in cipher_A psi. If a "canonical k" means k=0, then this is fine. But if the canonical k depends on the input (e.g., is pseudorandom), the composition may use a different k in each application, breaking the equalities.

**Suggestion**: Specify explicitly that the canonical k is a fixed constant (e.g., k=0) for the purposes of the functoriality proof, and note that the oblivious (pseudorandom k) version requires the quotient-category framing.

3. **Fix Example 2.2.** The encoding multiplicity argument is broken because g^{k + j(p-1)} = g^k in Z_p*. Use ElGamal-style (g^r, a*h^r) pairs as the multi-representation encoding. This directly motivates the ElGamal citation and is mathematically correct.

6. **Add a worked numerical example for Algorithm 1.** Walk through a small case: M = {("cat", 0), ("dog", 1), ("fish", 0)}, a 2-bit code (0->00, 1->01), and a toy hash function. Show the seed search finding l=3 (or whatever), the XOR computation, and verification. This is the clearest way to make Algorithm 1 concrete and will expose any remaining bugs in the description.

For probabilistic data structures + information theory: IEEE Transactions on Information Theory or ISAAC/STACS.
